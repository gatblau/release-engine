# Release Engine — Design (Part 4)

## 9) Design Contracts

- **Release Engine**: Authenticate/authorise intake, per-tenant rate limiting, idempotent writes, transactional projection updates, deterministic module and connector resolution, and two metrics surfaces by design.
- **Registry**: Single resolution plane for modules and connectors. Selection keys: path_key, provider, tenant_id, policy, region.
- **Module**: Orchestration only; no direct external calls. Emits transitions and timings to engine metrics interfaces.
- **Connector**: Protocol and auth boundary to an external service; swappable via registry without engine changes.
- **Metrics schema**:
    - SQL tables: jobs and steps with job_id, path_key, step_key, status, started_at, finished_at, duration_ms.
    - Prometheus metrics: DORA/SPACE families and per-step counters/histograms with bounded label sets.
- **Observability flow**: Prometheus scrapes exporter; remote_write to VictoriaMetrics; Grafana queries both SQL and PromQL.

---

## 10) Extension Path


- Add a new golden path: add a module and registry entry.
- Connectors are reused or added.
- No changes to the engine or metrics pipeline.

## 11) HTTP API Surface


All endpoints require JWT bearer from the OIDC provider. Authorization uses RBAC on (tenant_id, path_key, action). All timestamps are UTC. Maximum request body size: 256 KB.

### Health Endpoints

- **GET /healthz** — Process liveness. Returns 200 if the process is running.
- **GET /readyz** — Readiness. Returns 200 only when the database connection is healthy, PgBouncer is reachable, schema migrations are current, and the scheduler loop is running. Returns 503 otherwise.

### Job Endpoints

- **POST /v1/jobs**
    - Purpose: Deterministic, idempotent job submission.
    - Request body: `{ tenant_id, path_key, params, idempotency_key, callback_url? }`  (max 256 KB)
    - Validation: `callback_url` is validated to prevent SSRF attacks.
    - Rate limiting: Per-tenant token bucket. `429 Too Many Requests` with `Retry-After` header on exhaustion.
    - Response (deterministic per idempotency_key):
        - 202 Accepted — Body (canonical envelope): `{ job_id, tenant_id, path_key, state, attempt, max_attempts, next_run_at, accepted_at }`
        - 409 Conflict — Body: `{ error_code: "ERR_IDEM_CONFLICT", conflict: true, canonical: {...}, message: "idempotency key reused with different payload" }`
        - 413 Request Entity Too Large — Body: `{ error_code: "ERR_PAYLOAD_TOO_LARGE" }`
        - 429 Too Many Requests — Body: `{ error_code: "ERR_RATE_LIMITED" }`, Header: `Retry-After: <seconds>`
    - Headers (always same for a given key):
        - `Idempotency-Key`: echo of request key
        - `Idempotency-Replayed`: "true" when replay, "false" on first write
        - `Location`: /v1/jobs/{job_id}
    - Semantics:
        - Scope: (tenant_id, path_key, idempotency_key).
        - Exact payload match → return stored status and body bit-for-bit.
        - Payload mismatch → 409 Conflict; return canonical envelope; do not change state.
        - The canonical response status for a key is fixed at first write (202 Accepted).
        `idempotency_key` (string, required): Client-provided key. **Max 128 characters. Allowed characters: `[a-zA-Z0-9\-_.]`.** Keys outside this constraint return `400 Bad Request` with `error_code: ERR_INVALID_IDEMPOTENCY_KEY`.

- **GET /v1/jobs/{job_id}**
    - Purpose: Read current status from jobs_read.
    - Response: 200 OK with `{ job_id, tenant_id, state, attempt, owner_id, lease_expires_at, next_run_at, started_at, finished_at, updated_at, last_error_code?, last_error_message? }`

- **POST /v1/jobs/{job_id}/cancel**
    - Purpose: Cooperative cancellation request.
    - Authorization: Requires `job:cancel`. RBAC cache is **bypassed** for this action — fresh evaluation always.
    - Response:
        - 202 Accepted if cancel was requested or job is already terminal.
        - 200 OK if transition to cancelled occurred synchronously (state = queued).
    - Semantics: Sets cancel_requested_at and reason. If state = queued, transitions to cancelled immediately. If running, runner observes cancel and finalises to `cancelled`.

- **POST /v1/jobs/{job_id}/steps/{step_id}/decisions**
    - Purpose: Record an approval decision for a `waiting_approval` step.
    - Request body: `{ decision, justification?, idempotency_key?, run_id? }` where `decision` is one of `approved` or `rejected`.
    - Authentication and authorisation:
        - Approver identity comes from JWT subject.
        - Approver role is read from JWT role claim.
        - Policy evaluation enforces role allow-list, self-approval restriction, tenant scope, and optional budget authority.
    - Response:
        - 200 OK when decision is recorded.
        - 200 OK for idempotent replay with the same `idempotency_key` and payload.
        - 404 Not Found when job or step does not exist.
        - 409 Conflict when step is no longer waiting for approval, approver already decided, or idempotency key is reused with a different payload.
        - 403 Forbidden when approver role is not allowed.
        - 422 Unprocessable Entity when policy is violated (for example, self-approval).
    - Semantics:
        - If `idempotency_key` is omitted, key is derived as `sha256(approver + step_id + run_id)`.
        - Decision updates step progression atomically with decision persistence.

#### 11.1 Approval Guardrails Policy Contract

Approval decisions are evaluated by `PolicyEngine.EvaluateApproval()` using a dedicated policy input contract.

```go
type ApprovalPolicyInput struct {
    Approver         string            `json:"approver"`
    ApproverRole     string            `json:"approver_role"`
    ApproverTenantID string            `json:"approver_tenant_id"`
    JobOwner         string            `json:"job_owner"`
    JobTenantID      string            `json:"job_tenant_id"`
    PathKey          string            `json:"path_key"`
    StepKey          string            `json:"step_key"`
    BlastRadius      string            `json:"blast_radius"`
    Metadata         map[string]string `json:"metadata"`
    AllowedRoles     []string          `json:"allowed_roles"`
    SelfApproval     bool              `json:"self_approval"`
    MinApprovers     int               `json:"min_approvers"`
}

type ApprovalPolicyResult struct {
    Allowed    bool     `json:"allowed"`
    Reason     string   `json:"reason"`
    Violations []string `json:"violations"`
}
```

Built-in approval guardrails:

- **Self-approval block**: denied when `self_approval=false` and `approver == job_owner`.
- **Role allow-list**: denied when approver role is not in `allowed_roles`.
- **Tenant scope**: denied when `approver_tenant_id != job_tenant_id`.
- **Budget authority** (optional): denied when `metadata.estimated_cost > metadata.approver_limit`.
- **Four-eyes progression**: step remains in `waiting_approval` until `min_approvers` approved decisions are recorded.

Evaluation semantics:

- `EvaluateApproval()` is a **pure authorisation decision** for a single submitted decision.
- Four-eyes threshold progression is enforced by `ApprovalService` after each accepted decision.
- Distinct approvers are counted by identity (`approver`) and cannot submit more than one decision per step.

- **GET /v1/jobs**
    - Purpose: Query jobs, including pending approval workload.
    - Query parameters:
        - `step_status` (optional): when set to `waiting_approval`, returns jobs with pending approvals.
        - `tenant` (optional): tenant filter.
        - `approver_role` (optional): only return approvals eligible for this role.
    - Response:
        - 200 OK with `{ jobs: [...] }`.

- **GET /v1/jobs/{job_id}/steps/{step_id}/approval-context**
    - Purpose: Fetch read-only approval context for UI clients.
    - Response includes:
        - `approval_request` context (summary, detail, blast_radius, policy_ref, metadata)
        - policy requirements (`min_approvers`, `allowed_roles`, `self_approval`)
        - decision tally and recorded decisions
        - optional expiry timestamp
    - Response codes:
        - 200 OK on success.
        - 404 Not Found when job or step is unknown.

- **GET /v1/jobs/{job_id}/events** (optional)
    - Purpose: Server-sent events or streaming over HTTP/2 with last-event-id cursor.
    - Response: text/event-stream. Events derived from outbox with per-tenant cursor.

### Webhooks / Callbacks

Outbound POST to callback_url with signed HMAC header and replay protection via delivery_id. At-least-once delivery.

#### Approval Outbox Event Contracts

Approval lifecycle events are first-class outbox contracts and are emitted transactionally with their corresponding state change.

| Event Type | Emitted By | Trigger | Required Payload Fields |
|---|---|---|---|
| `approval_requested` | `ApprovalService` | Step enters `waiting_approval` | `job_id`, `step_id`, `approval_request`, `policy_ref`, `expires_at` |
| `approval_decided` | `ApprovalService` | Human decision accepted | `job_id`, `step_id`, `decision`, `approver`, `justification` |
| `approval_escalated` | `ApprovalWorker`/`ApprovalService.TickApprovals` | Elapsed reaches `escalation_at * ttl` | `job_id`, `step_id`, `elapsed_seconds`, `remaining_seconds`, `approval_request` |
| `approval_expired` | `ApprovalWorker`/`ApprovalService.TickApprovals` | TTL exceeded and step auto-failed | `job_id`, `step_id`, `reason=approval_timeout`, `original_request`, `ttl_seconds` |

Contract rules:

- Event type must be registered in `OutboxDispatcher` before emission.
- Unknown/unregistered event types are rejected at enqueue time.
- Event payloads are immutable once queued.
- Approval events follow standard outbox retry/DLQ behaviour.

**Outbox Delivery Guarantees**:
- **Ordering**: FIFO per job is not guaranteed. Events may be delivered out of order across different jobs. For a single job, events are delivered in submission order.
- **Max delivery attempts**: Each outbox entry has `max_attempts` defaulting to **12**. After exhausting attempts, the entry is promoted to `dlq` status and an alert fires.
- **Backoff schedule**: Exponential backoff with jitter: `LEAST(2^(attempt-1), 600)` seconds (capped at 10 minutes).
- **Delivery timeout**: HTTP requests have a **10-second** timeout. Timeout or connection errors are treated as transient failures and retried.
**DLQ promotion**: When `attempt >= max_attempts`, `delivery_state` is set to `'dlq'`. Manual intervention is required to replay or discard DLQ entries.
**4xx response handling**: HTTP 4xx responses from the callback endpoint are treated as permanent client-side rejections. The outbox entry is marked `delivered` without incrementing `attempt`, and the HTTP status/body are captured in `last_error` for audit. 5xx responses and network errors continue to follow the retry schedule.
**Idempotency contract**: Consumers MUST handle duplicate deliveries. Each delivery includes a unique `delivery_id` in the payload. Consumers should track processed `delivery_id` values to detect and ignore duplicates.

**HMAC Key Rotation**: Two signing keys are maintained per tenant during rotation. New events are signed with the active key; receivers verify against both keys. The `X-Signature-Version` header identifies which key was used. A key rotation runbook is maintained in the operations' playbook.

**Notes**
- The API never returns 200 OK for POST /v1/jobs. 202 is stable and communicates asynchronous processing.
- Clients that need to know whether a response was a replay read the `Idempotency-Replayed` header.

## 12) Data Model (Postgres)

One primary Postgres cluster for jobs, projection, and outbox. All instances connect through **PgBouncer in transaction mode** to share the connection budget efficiently.

> **Constraint: No session-level advisory locks.** Transaction-mode PgBouncer proxies connections at the transaction boundary, which means any session-scoped state—including `pg_advisory_lock`—is reset between transactions. If advisory locks are needed for coordination, use transaction-level `pg_advisory_xact_lock()` instead. Using session-level `pg_advisory_lock()` will silently fail under PgBouncer.

Connection budget guidance:
- Target 10–30 active connections per instance from the app pool.
- Size PgBouncer `max_client_conn` to `instances × workers_per_instance × safety_factor`.
- Alert on pool exhaustion (`pgbouncer_cl_waiting > 0` for > 30 s).

TimescaleDB (extension or separate cluster) stores metrics_sql.

### 12.5 external_effects

> **Note**: `external_effects` tracks **connector calls** only — the durable record of every outbound API call to an
> external provider. Outbound webhook/callback deliveries to tenant-registered `callback_url` endpoints are managed
> exclusively by the `outbox` table (§12.3). The `effect_kind` enum therefore contains only `'connector_call'`.

```sql
-- Only connector calls are tracked here; webhook/callback delivery is handled by the outbox table (§12.3).
CREATE TYPE effect_kind AS ENUM ('connector_call');

CREATE TYPE effect_status AS ENUM (
  'pending',       -- awaiting reservation by runner
  'reserved',      -- reservation acquired, awaiting execution
  'in_flight',     -- currently executing
  'succeeded',     -- completed successfully
  'failed',        -- terminal failure
  'canceled',      -- cancelled by job cancellation
  'unknown_outcome', -- network timeout or ambiguous response
  'dlq'            -- escalation after max reconciliation attempts
);

CREATE TABLE external_effects (
  effect_id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  kind              effect_kind NOT NULL DEFAULT 'connector_call',
  job_id            uuid NOT NULL,
  run_id            uuid NOT NULL,   -- matches jobs.run_id type; used for fencing
  step_key          text NOT NULL,
  connector_key     text,
  operation         text NOT NULL,
  input_digest      bytea NOT NULL,
  call_id           text NOT NULL,
  status            effect_status NOT NULL DEFAULT 'pending',
  attempt           int NOT NULL DEFAULT 0,
  max_attempts      int NOT NULL DEFAULT 3,       -- effect-level retry limit
  reconcile_attempts int NOT NULL DEFAULT 0,      -- tracks reconciliation cycles for DLQ escalation
  max_reconcile_attempts int NOT NULL DEFAULT 5,  -- reconciler retry budget before DLQ escalation
  owner_id          text,
  lease_expires_at  timestamptz,
  next_run_at       timestamptz,
  provider_ref      text,
  last_error_code   text,
  last_error_message text,
  created_at        timestamptz NOT NULL DEFAULT now(),
  updated_at        timestamptz NOT NULL DEFAULT now(),
  UNIQUE (job_id, run_id, step_key, operation, input_digest),
  UNIQUE (call_id)
);

CREATE INDEX ON external_effects (status, next_run_at) WHERE status IN ('reserved','unknown_outcome');
CREATE INDEX ON external_effects (status, lease_expires_at) WHERE status = 'in_flight';
CREATE INDEX ON external_effects (status) WHERE status = 'dlq';  -- alert scanning
```

Helper functions:

```sql
-- Returns true if the lease was successfully acquired; false if the row was not in 'reserved' state
-- (i.e. another worker claimed it first). Callers MUST check the boolean return value.
CREATE OR REPLACE FUNCTION effect_lease(_effect_id uuid, _owner text, _ttl_secs int)
RETURNS boolean LANGUAGE plpgsql AS $$
DECLARE
  _updated int;
BEGIN
  UPDATE external_effects
     SET status = 'in_flight',
         owner_id = _owner,
         lease_expires_at = now() + make_interval(secs => _ttl_secs),
         updated_at = now()
   WHERE effect_id = _effect_id
     AND status = 'reserved';
  GET DIAGNOSTICS _updated = ROW_COUNT;
  RETURN _updated = 1;
END;
$$;

CREATE OR REPLACE FUNCTION effect_backoff(_effect_id uuid, _policy jsonb, _code text, _msg text)
RETURNS void LANGUAGE plpgsql AS $$
BEGIN
  UPDATE external_effects
     SET status = 'reserved',
         attempt = attempt + 1,
         owner_id = NULL,
         lease_expires_at = NULL,
         next_run_at = now() + backoff_interval(attempt, _policy),
         last_error_code = _code,
         last_error_message = _msg,
         updated_at = now()
   WHERE effect_id = _effect_id
     AND status = 'in_flight';
END;
$$;

-- Escalate unknown_outcome to dlq after max reconcile attempts.
-- Uses the per-row max_reconcile_attempts column (not a parameter) to honour per-effect configuration.
CREATE OR REPLACE FUNCTION effect_escalate_dlq(_effect_id uuid)
RETURNS void LANGUAGE plpgsql AS $$
BEGIN
  UPDATE external_effects
     SET reconcile_attempts = reconcile_attempts + 1,
         status = CASE
                    WHEN reconcile_attempts + 1 >= max_reconcile_attempts THEN 'dlq'
                    ELSE 'unknown_outcome'
                  END,
         next_run_at = CASE
                         WHEN reconcile_attempts + 1 >= max_reconcile_attempts THEN NULL
                         ELSE now() + interval '5 minutes'
                       END,
         updated_at = now()
   WHERE effect_id = _effect_id
     AND status = 'unknown_outcome';
END;
$$;
```

### 12.6 steps

Tracks per-attempt step completions for a job. Used by the Runner on retry to skip already-succeeded steps
(`BeginStep` is a no-op for steps that already have a `status='ok'` row for the current attempt chain).

```sql
CREATE TYPE step_status AS ENUM ('ok', 'error', 'skipped', 'waiting_approval');

CREATE TABLE steps (
  id           bigserial PRIMARY KEY,
  job_id       uuid        NOT NULL REFERENCES jobs (id) ON DELETE CASCADE,
  run_id       uuid        NOT NULL,   -- run_id at time of recording; used for fencing
  attempt      int         NOT NULL,
  step_key     text        NOT NULL,
  status       step_status NOT NULL,
  output_json  jsonb,                  -- populated on status='ok'
  error_code   text,                  -- populated on status='error'
  error_message text,                 -- populated on status='error'
  started_at   timestamptz NOT NULL DEFAULT now(),
  finished_at  timestamptz,
  duration_ms  int,
  approval_request jsonb,              -- ApprovalRequest payload
  approval_ttl     interval,           -- optional timeout
  approval_expires_at timestamptz,      -- computed from ttl
  UNIQUE (job_id, attempt, step_key)  -- one row per step per attempt
);

-- Efficient lookup for step resumption on retry
CREATE INDEX steps_job_attempt_idx ON steps (job_id, attempt);
```

**Step resumption invariant**: On retry, the Runner queries `steps` for all rows matching `(job_id, attempt - 1)` where `status = 'ok'`. `BeginStep` returns immediately (no-op) for any step found there, allowing modules to be written as straight-line code.

### 12.7 job_context

Persists transient job-scoped key/value pairs set by modules via `StepAPI.SetContext()`. Values survive retries and

### 12.10 Intake Transaction (Deterministic Envelope)

```sql
BEGIN;

INSERT INTO idempotency_keys AS ik(
  tenant_id, path_key, idempotency_key, payload_fingerprint,
  status_code, response_body_json, job_id, accepted_at, expires_at
)
VALUES (
  $tenant, $path, $idem_key, $fingerprint,
  202, '{}'::jsonb, '00000000-0000-0000-0000-000000000000', now(),
  now() + interval '48 hours'
)
ON CONFLICT (tenant_id, path_key, idempotency_key)
DO NOTHING;

-- If inserted row_count = 1, we own creation:
UPDATE idempotency_keys
   SET response_body_json = $envelope_json,
       job_id = $job_id,
       updated_at = now()
 WHERE tenant_id = $tenant
   AND path_key = $path
   AND idempotency_key = $idem_key;

COMMIT;
```

Conflict check:

```sql
SELECT payload_fingerprint, status_code, response_body_json
  FROM idempotency_keys
 WHERE tenant_id = $tenant
   AND path_key = $path
   AND idempotency_key = $idem_key
 FOR SHARE;
```
